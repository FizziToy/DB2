# Відповіді на завдання

## 1) Види NoSQL баз

Існує чотири основні категорії NoSQL баз даних:

## Key-Value Stores
|Тип | Бази даних "Ключ-значення" (Key-Value Stores)|
| :--- | :--- |
|Принцип| Дані зберігаються як набір унікальних ключів та пов'язаних з ними значень|
|Особливості| Ключ використовується для швидкого отримання значення. Значення може бути простим об'єктом або складним складовим об'єктом|
|Використання| Кешування, сесії користувачів, кошики для покупок (e-commerce), ігрові та IoT-додатки|
|Приклади| Redis, Riak, Amazon DynamoDB|

## Document Databases
|Тип | Документ-орієнтовані бази даних (Document Databases)|
| :--- | :--- |
| Принцип| Зберігають дані у вигляді напівструктурованих документів (зазвичай у форматі JSON, BSON або XML). Кожен документ може мати різну структуру|
|Особливості| Забезпечують гнучку схему, що дозволяє легко додавати або змінювати поля без впливу на інші документи. Добре підходять для складних, ієрархічних даних|
|Використання| Каталоги, системи керування вмістом, профілі користувачів, веб-додатки|
|Приклади| MongoDB, Couchbase, DocumentDB|

## Wide-Column Stores
|Тип | Колоночні (або Широко-колоночні) бази даних (Wide-Column Stores)|
| :--- | :--- |
|Принцип| Дані зберігаються в сімействах колонок, а не в рядках, як у реляційних БД. Кожен рядок може мати різний набір колонок.|
|Особливості| Дуже ефективні для зберігання величезних обсягів даних з багатьма атрибутами, оскільки оптимізовані для швидкого доступу до даних у стовпцях. Це забезпечує високу масштабованість і доступність.|
|Використання| Big Data, аналітика в реальному часі, лічильники, великі розподілені системи.|
|Приклади| Apache Cassandra, HBase.|

## Graph Databases
|Тип | Графові бази даних (Graph Databases)|
| :--- | :--- |
|Принцип| Використовують графову модель для зберігання даних. Дані представлені вузлами (сутностями), ребрами (зв'язками між вузлами) та властивостями (інформацією, пов'язаною з вузлами та ребрами).|
|Особливості| Ідеально підходять для моделювання та навігації складними взаємозв'язками, оскільки зв'язки зберігаються на рівні даних.|
|Використання| Соціальні мережі (зв'язки між друзями), системи рекомендацій, виявлення шахрайства, управління мережевою інфраструктурою.|
|Приклади| Neo4j, Amazon Neptune.|

Крім цих чотирьох основних типів, існують і інші, більш спеціалізовані NoSQL бази даних:
- Бази даних часових рядів (Time-Series Databases): Оптимізовані для зберігання та аналізу даних, які індексуються за часом (наприклад, показники датчиків IoT).
- Об'єктні бази даних (Object Databases): Зберігають дані як об'єкти, які використовуються в об'єктно-орієнтованому програмуванні.

## 2) Доцільність винесення частини даних в іншу базу**

**RealEstateDB: Архітектура та стратегія винесення даних**

Цей документ описує архітектурне рішення щодо стратегічного винесення некритичних та високомасштабованих компонентів системи управління нерухомістю (Property Management System) з основної реляційної бази даних (MSSQL) у спеціалізовані NoSQL сховища.

Основна реляційна база (MSSQL) зберігає критичні дані (Lease, Invoice, Payment, Tenant, Unit) для забезпечення ACID-транзакцій та цілісності даних.

Для деяких таблиць буде доцільно винесення певні типи даних у спеціалізовані NoSQL бази даних, а саме:

### Журнал аудиту (AuditLog)
|Поточна таблиця| `AuditLog` |
| :--- | :--- |
| **Тип NoSQL** | **Колоночна БД** або **Документна БД** |
| **Аргументи ефективності** | Це дані, що постійно додаються (write-heavy), які рідко оновлюються. Колоночна або Документна БД надає горизонтальну масштабованість та високу швидкість запису, що є ідеальним для масивного логування, дозволяючи не перевантажувати основну реляційну базу. |
| **Перевага** | Зниження навантаження на основний сервер MSSQL. |

### Показники лічильників (MeterReading)
|Поточна таблиця| `MeterReading` |
| :--- | :--- |
| **Тип NoSQL** | **База даних часових рядів** |
| **Аргументи ефективності** | Це класичні Timeseries-дані (мітка часу + значення). Спеціалізовані TSDB стискають дані, оптимізують запити на агрегацію за часовими інтервалами (наприклад, "середнє споживання за місяць") і значно економлять ресурси та підвищують швидкість аналізу порівняно з реляційною таблицею з мільйонами записів. |
| **Перевага** | Швидкий аналіз, ефективне зберігання та компресія. |

### Запити та замовлення (ServiceRequest та WorkOrder)
| Поточні таблиці | `ServiceRequest`, `WorkOrder` |
| :--- | :--- |
| **Тип NoSQL** | **Документна БД** |
| **Аргументи ефективності** | Ці сутності часто містять неструктуровані або напівструктуровані деталі (коментарі, журнали змін статусу, різні форми запитів). Документна БД дозволяє гнучко змінювати схему (додавати нові поля для різних типів запитів) та зберігати вкладені дані, забезпечуючи кращу гнучкість для швидких ітерацій розробки функціоналу обслуговування. |
| **Перевага** | Гнучкість схеми, швидке додавання нових полів без міграції. |

## 3) Обрана NoSQL база**
Обрала MongoDB та винесла в NoSQL дані про заявки орендарів (ServiceRequest).Це доцільно, тому що структура заявок може змінюватися, вони можуть містити історію, коментарі та вкладені дані.MongoDB дозволяє зберігати одну заявку як документ без складних JOIN та без необхідності змінювати структуру таблиць у SQL.
### Що реалізовано
- Створена база RealEstateNoSQL та колекція service_requests.
- Додані тестові документи через MongoDB Compass.
- Реалізовано підключення MongoDB у проєкті ASP.NET MVC.
- Створено сторінку, яка показує заявки із NoSQL-бази.
